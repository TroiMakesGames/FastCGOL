Multithreading can be achieved for even more optimisation, but a small change needs to be implemented first

currently the live irrelavance check uses an unordered set to store active cell coordinates without containing duplicates. In case of multithreading this is suboptimal as it introduces possibilities of race conditions between threads

A fix for this is very simple, but we have to take a step back. Instead of trying to remove duplicates from active coords we can simply let all threads push anything to the list and then in the next frame, where the newActiveCoords becomes activeCoords we simply make sure to not check a single cell multiple times

(basicaly instead of avoiding duplicates we skip duplicates in future checks)

This can be done with a simple flag array where we store which cells have been checked and which havent. When looping through the activeCoords, once we reach a copy of a previously checked active coord,  because we marked the coord as checked, the copy is simply skipped

This method is better for multithreading (as we allow overpushing and avoid race conditions) and we get the same result of skipping mutliple checks for a single active cell

Developers note after implementing the hasCellBeenProccessed flag array:
Since the datatype of a simple bool array is much more simple than an unordered set all of the removed computing required to not add copies to the unordered set made the simulation around 80% faster already, even tho the whole change was made to allow multithreading.

Developers note before implementing multithreading:
during multithreading since the activeCoords still contains duplicates, 2 threads may want to compute the same coordinate at the same time, which introduces race conditions.
Even tho both threads eventualy do come to the same result of the newGrid cell state and both write the same (True) value to the hasCellBeenProccessed flag array this still leaves possibilities for undefined states.

To fix this all that needs to change is that the hasCellBeenProccessed flag array becomes an atomic bool array, where reading and writting is done as an non-divisible (atomic) operation. Once a thread reads that a coordinate has not yet been proccesed, it will immediatly exchange the value from false to true before any other thread can read the state.
This means that any other thread will read that the coordinate cell has been/is being proccessed already.

The same problem would occur with the newGrid array in case both threads computed a coordinate and wanted to add it to the newGrid, but since this happens as a final part of the whole coordinate operation its already fixed by the hasCellBeenProccessed check, meaning that only one thread is ever allowed to compute that newGrid index. 

The atomic bool array is a more complex data structure than the previous simple bool array which means that the performance naturally drops between normal flagged and atomic flagged, with the hope that multithreaded will perform even better than normal flagged.

Developers note after implementing multithreading:
The resulting performance of multithreading is worse than expected, but interesting none the less. Compared to the best performing previous version (licgol_flagged) it reaches a lower highest FPS but surprisingly it reaches its highest FPS much faster, which may or may not result in an overall faster computation of the whole simulation up until an ending state (no more activity in the CGOL simulation past oscilating)

Also i havent really tested this correctly but it seems that the multithreading version is better capable of computing at higher world resolutions.
